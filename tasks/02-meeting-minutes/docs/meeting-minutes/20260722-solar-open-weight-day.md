## 3. 세션별 요약

### 3.1 Solar Open 2 소개 — Sung Kim (Upstage CEO)

**핵심 메시지**: "벤치마크 점수 경쟁이 아닌, 실용성과 에이전트 능력에 집중한 모델"

- **개발 배경**: 독파모 국책 사업 지원을 통해 250B 파라미터 모델 개발 실현 (이전 Solar Open 1: 100B)
- **핵심 설계 철학**:
  - 단순 벤치마크 점수 경쟁 지양 → **실용성, 높은 지능, 에이전트 툴 사용 능력**에 집중
  - **한국어 압도적 성능**: 정부 과제를 통한 대규모 한국 데이터 확보·활용
- **경쟁력**: Mistral, Qwen, Gemma 등 동급 모델 대비 우수. 특히 Live Code Bench 및 에이전트 성능에서 Solar Open 1 대비 "너무너무 작아 보일 정도"로 향상
- **실용성 강점**: 경쟁 대형 모델(2.8T~2.4T급)은 GPU 16장 이상 + 월 1억 원 이상 소요 vs **Solar Open 2는 B200 2장으로 구동 가능**
- **사용자 반응**: 현업으로부터 "GLM 5.2보다 좋다", "가슴을 널뛰게 하는 좋은 모델" 평가 수령
- **배포**: 허깅페이스에서 풀 모델 및 양자화 모델 즉시 다운로드 가능. API도 개방 (미신청자는 가입 후 신청 시 최대한 개방 예정)

**신규 개념 — Office-verse (오피스 버스)**:
소프트웨어 유니버스(시뮬레이션 환경)에서 실제 공개 데이터를 활용해 과제를 학습시키고, 엑셀·문서·PPT 등 실제 산출물을 생성한 뒤 자동 채점을 통해 학습 데이터를 구축하는 파이프라인. 11개 산업 분야, B2B 과제 유형에 특화. Claude가 코드 분야에서 두각을 나타낸 것처럼, **사무 환경 에이전트 분야에서 Solar가 선도적 위치 확보**를 목표. GPT 계열 대비 모델 사이즈가 작음에도 오피스 환경 태스크에서 **압도적 성능** 달성.

### 3.2 Solar Open 2 기술 소개 — Hwalsuk Lee (Upstage CTO)

**핵심 메시지**: "하이브리드 어텐션, NoPE, 가중치 이식, RL 중심 사후학습 — 3가지 핵심 기술 역량 확보"

#### (1) 사전 학습(Pre-training) 구조 혁신

**하이브리드 어텐션 설계**:
- Open 1: 전체 Softmax 어텐션 → Open 2: **Softmax + Linear 어텐션 3:1 혼합**
- Softmax: 롱 컨텍스트 정확도 높으나 메모리·연산량 급증
- Linear: 메모리·연산 효율 우수하나 롱 컨텍스트 정확도 담보 어려움
- **결과**: 메모리 소비·연산량 기존 대비 약 1/4 수준으로 감소하면서도 정확도 유지

**RoPE 제거 및 NoPE 설계**:
- 기존 널리 사용되던 **RoPE(Rotary Position Embedding)를 완전히 제거**
- 모델이 스스로 포지셔닝을 학습하도록 설계
- **보완 장치**: (a) 시그모이드 게이트 추가 — 리니어 스테이트의 토큰 순서 자체 학습 안정화, (b) 네거티브 아이겐밸류 도입 — 리니어 어텐션에서 음수 아이겐밸류 허용하여 이전 값 삭제/수정 학습 가능
- 이론적 무한대 컨텍스트 지원, 공식 지원 범위 **1M 토큰 이상**

**구조 효율성 검증 (Ablation Study)**:
- 10B 프록시 모델로 어블레이션 스터디 수행
- 동일 크기·데이터 조건에서 제안 구조 적용 시, 기존 구조 671B 토큰 도달 성능을 **210B 토큰 내에 달성 (약 3배 학습 효율)**

#### (2) 모델 초기화 및 가중치 이식 기술

- Open 1의 학습 결과를 최대한 활용하기 위해 **랜덤 초기화 대신 가중치 이식(Weight Transfer) 방식 채택**
- 어텐션 구조가 완전히 변경되어 직접 매핑 불가 → 이식 가능/재매핑 필요 파라미터 실험적 구분
- 전체 파라미터(250B) 중 약 **2.3%에 해당하는 5.7B 파라미터만 이식**, 나머지는 랜덤 초기화
- 100B 프록시 모델 실험에서 17B 토큰 소비 시점 기준 이식 초기화가 랜덤 초기화 대비 **현저히 높은 성능 수렴 속도** 입증
- 향후 Open 3 개발 시에도 동일 적용 예정 (Depth-Up Scaling 등 초기화 기술 노하우 축적 결과)

#### (3) 사후 학습(Post-training): RL 중심 전략

- Open 1: SFT → DPO 방식 → Open 2: **SFT → RL(강화학습) 방식 전환** (프론티어 랩 추세 추종)
- 프론티어 랩들은 RL 단계에 풀 트레이닝보다 훨씬 많은 GPU 투입, GPU 스케일 확장에 비례해 성능 향상
- **도메인 특화 티처 모델**: 코딩, 오피스 워크 등 **도메인별 12개 특화 모델**을 각각 SFT + RL로 학습하여 티처 모델로 활용
- **온-폴리시 디스틸레이션(On-policy Distillation)**을 통해 대규모 RL 수행 → 최종 성능 확보
- **Verifiable 데이터 생성 파이프라인**: RL의 핵심 관문. 각 도메인별 에이전트 작동 가상 환경 별도 구축 (GPU + CPU 대규모 필요). 오피스 워크 환경: 초기 170개 태스크 → HWP 지원 추가 등 지속 고도화
- **비동기(Async) RL 인프라**: 동기 방식 초기 GPU 활용률 20%에 불과 → **비동기 방식으로 전환**, 다양한 리워드 시그널을 효율적으로 합산하여 배치 학습에 투입하는 자체 RL 방법론 개발

#### (4) 성능 결과 및 향후 계획

- 에이전트형 벤치마크에서 Open 1 대비 크게 향상, 한국어 벤치마크 전반 우수
- **GDPBench(한국형 자체 버전)** 평가에서 유사 규모 최신 모델들 대비 높은 성능
- 현재 릴리즈 모델: RL 충분히 수행하지 않아 **추론(reasoning) 과정이 다소 길게 출력**되는 경향 → 개선된 버전 곧 출시 예정
- **Solar Open 2 Pro** 버전 준비 중 (더 높은 성능 기대)
- 온프레미스 서빙 시 **양자화(Quantization) 활용 권장**

### 3.3 MoE 최적화를 통한 GPU 절감 — Taeho Kim (NotaAI CTO)

**핵심 메시지**: "MoE 모델의 양자화·프루닝 취약점을 해결하는 3가지 기술 — GPU 2장으로 서빙 가능"

#### MoE 모델 최적화의 배경

Solar Open 2는 MoE(Mixture of Experts) 기반 모델로, MoE 구조는 특히 **양자화(Quantization)와 오프로딩(Offloading)에 취약**하다는 문제가 있음. 이를 해결하기 위해 NotaAI가 개발한 세 가지 핵심 기술 소개.

#### 세 가지 핵심 기술

**① Parscal MoE — 캘리브레이션 데이터셋 균형화**:
- 양자화 수행 시 캘리브레이션 데이터셋 확보 필수
- MoE 모델에서 일반적 데이터 수집 방식으로는 특정 Expert에 데이터 몰리고 일부 Expert는 데이터 부족해지는 **분포 불균형 문제** 발생
- **해결**: 모든 Expert에 고르게 데이터가 분포되도록 하는 기술 개발
- 리모트 해커톤 대회 1등 수상 경험 기반으로 Solar Open 2에 적용

**② 라우터 의사결정 보존 — 추가 Loss Function 설계**:
- 양자화 후 MoE 모델 성능 저하의 핵심 원인: **라우팅 의사결정의 왜곡**
- 레이어가 깊어질수록 원본 모델과 양자화 모델 간 라우팅 일치도가 급격히 하락하는 현상 확인
- **해결**: 라우터의 의사결정이 틀어지지 않도록 하는 **추가 Loss Function 설계**
- 결과: NotaAI 알고리즘(그래프상 빨간색)은 레이어 깊이와 무관하게 높은 라우팅 일치도 유지
- 관련 논문 2편 **ICML AdaptFM 워크숍**(양자화 최적화 특화)에 게재 (1편 자체 발표, 1편 Upstage와 공동 발표)

**③ 글로벌 Expert 프루닝 — 레이어별 비균일 제거**:
- MoE 모델에서 일부 Expert만 활성화되므로 중요도가 낮은 Expert 제거(프루닝) 가능
- 일반적 방식: 레이어마다 균일하게 Expert 제거
- **NotaAI 방식**: 전체 레이어·Expert 통합 분석하여 **레이어별로 제거 비율을 차등 적용**하는 글로벌 프루닝 기술
- 결과: 정확도 손실 없이 모델 크기 축소 가능

#### 적용 결과

- BF16 원본 모델(약 500GB) 대비:
  - **INT4: 약 140GB** (약 72% 감소)
  - **MX FP4: 약 153GB** (약 70% 감소)
- 컨텍스트 길이 256K 토큰 기준:
  - INT4 단순 양자화 시 GPU 2장 초과 메모리 필요 (사실상 4장 요구)
  - **약 20% 프루닝 적용 시 GPU 2장으로 서빙 가능**
- 성능: BF16 원본 대비 양자화 모델 모두 높은 성능 유지
  - 특히 SOTA 프루닝 알고리즘인 **LLM-Pruner(Leaf) 대비 추론(Reasoning) 태스크에서 현저히 우수**한 성능

#### 권장 사항

- Blackwell 또는 MX FP4 지원 디바이스 → **MX FP4** 사용 권장
- Hopper 등 INT4 친화적 환경 → **INT4** 사용 권장

### 3.4 활용 사례 시연 — 이태호 (Upstage 사업 개발)

**핵심 메시지**: "코딩을 한 줄도 배운 적 없는 비개발자가 3D 게임을 개발·배포 — Solar Open 2의 실용성 실증"

- **발표자 배경**: 이태호는 Upstage 사업 개발 담당으로, **코드를 한 줄도 배운 적 없는 비개발자**임을 직접 밝힘
- **데모 내용**: Solar Open 2 + Claude Code로 **3D 양궁 게임** 개발·시연
- **기획 의도**: "대한민국 하면 국가대표 양궁"이라는 단순·직관적 발상. 실용성과 시각화가 가장 중요하다고 판단
- **기술 구현**:
  - Three.js(3JS) 라이브러리 활용
  - 화살 포물선 궤적, 과녁 적중 시 점수 판정, 3D 수학 연산 등 비개발자에게 생소한 요소 다수 포함
  - **원샷이 아닌 수차례의 반복 수정**을 통해 완성도 높은 게임 구현
  - 화살 불타오르는 시각 효과 추가 수정도 데모 중 직접 시연

## 5. 종합 인사이트

### Solar Open 2의 3대 핵심 경쟁력

1. **실용성**: B200 2장으로 구동 가능한 효율성. 2.4T~2.8T급 경쟁 모델이 GPU 16장+월 1억 원 이상 소요되는 것과 대비
2. **한국어·오피스 워크 에이전트**: 정부 과제 기반 대규모 한국 데이터 확보. OfficeVerse 파이프라인으로 사무 환경 에이전트 분야에서 선두 경쟁력 확보
3. **구조적 혁신**: 하이브리드 어텐션(3:1)으로 메모리·연산 1/4 감소, NoPE로 1M 컨텍스트 지원, 가중치 이식으로 학습 효율 3배 향상, RL 중심 사후학습으로 에이전트 성능 극대화

### 생태계 확장 전략

- **프리미엄(Freemium)**: 오픈웨이트로 생태계 확장 → 고도화 제품으로 유도
- **커뮤니티 선순환**: 정우준(HF)의 통찰 — "성능은 기본, 차별화 요소는 다양한 유즈 케이스와 따라 하기 쉬운 코드·레시피의 공개"
- **온프레미스 지원**: 법률·공공 등 보안 민감 분야의 AI 도입 장벽 해소
